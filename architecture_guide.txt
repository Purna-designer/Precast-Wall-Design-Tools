Purpose of This DocumentThis document bridges the gap between the React web application built as a UI prototype for the Precast WALL Designer and the equivalent application you could build in WPF with C# and the MVVM pattern. It covers three things:Data models — every object the application tracks, written as C# classes.Data flow — how data travels from ETABS import through design calculation to the summary sheet, in both the web and WPF worlds.Architecture comparison — React components and state hooks mapped side-by-side to ViewModels, Models, Commands, and bindings.After reading this you will understand:Why the web prototype keeps everything in memory (and why that is fine for a UI prototype).Which parts of the web code map directly to WPF ViewModel properties and which need new infrastructure.What a production-quality WPF version would look like at the folder / class level.2.  Application Overview2.1 What the Application DoesThe application replaces a multi-sheet Excel workbook that structural engineers used to design precast reinforced-concrete Walls per IS 456. The original spreadsheet required the engineer to:Paste ETABS forces manually into a Forces sheet.Press six buttons in a fixed order to generate two interaction curves (major and minor axis).Check a SAFE/UNSAFE cell after the fact.Manually copy results to a Summary tab before moving to the next wall.The new application makes every step visible, interactive, and persistent across multiple walls and projects simultaneously. The centrepiece is a live P-M interaction diagram that redraws in real time when the engineer adjusts the steel percentage (Pt), so the consequence of every decision is seen immediately.2.2 The Three ScreensScreenWPF EquivalentPurposeDashboardMainWindow.xaml / ProjectsView.xamlLists all projects with summary health (safe / review / unsafe counts).Project ViewProjectWindow.xaml / WallsListView.xamlShows all Wall members in one project — filterable, sortable.Design WorkspaceDesignWorkspace.xaml (main window)Full design flow for a single Wall: geometry → loads → interaction diagram → finalise.3. Both implementations follow an identical logical chain:Engineer moves slider      │      ▼ Pt value changes   (React: setPt(value)  |  WPF: Slider.Value → TwoWay binding → Pt property setter)      │      ▼ Curve recalculates  (React: useMemo rerun  |  WPF: OnPropertyChanged(CurvePoints) fires)      │      ▼ Demand point re-evaluated against new curve  (ratio = demandM / capacityAt(curve, demandP))      │      ▼ IsSafe recomputed   (ratio <= 1.0)      │      ▼ UI updates: curve SVG redraws, SAFE/UNSAFE badge changes colour, Apply button appears/disappears      │ (only on 'Add to Summary' click)      ▼ Pt written back to WallMember model and persistedWhy no database call on every slider move?  Both implementations are deliberate about NOT saving on every change. The curve calculation is purely computational — it runs in memory. Saving to disk/database only happens when the engineer explicitly confirms (Add to Summary). This keeps the UI fast and gives the engineer freedom to experiment.7.  The Interaction Curve Engine7.1 What the Engine DoesThe generateInteractionCurve() function in the web app (originally CommandButton1_Click in the spreadsheet's VBA) produces the P-M capacity envelope using the IS 456 strain-compatibility method. It:Sweeps the neutral axis depth from near-zero to the full section depth in 60 steps.For each neutral axis position, calculates the concrete stress block force and centroid using IS 456 parabolic-rectangular coefficients (0.362, 0.416, 0.447).For each bar layer, calculates strain, looks up the IS 456 design stress-strain table for Fe415/Fe500, and computes the steel force and its moment about the centroid.Stores the result as a normalised point: Pu/(fck·B·D) on the y-axis, Mu/(fck·B·D²) on the x-axis.The set of 61 normalised points forms the interaction envelope. A demand point that falls inside the envelope is SAFE; one that falls outside is UNSAFE.7.2 Mapping to a WPF Service ClassIn WPF this calculation lives in a Service class — not in the ViewModel. The ViewModel calls the service; the service returns a result object. This keeps calculation logic testable and separate from UI concerns.// C# — Services/InteractionCurveService.cspublic class InteractionCurveService{    // Mirrors generateInteractionCurve() in the web app    public InteractionCurveResult Generate(        WallMember col,        Axis axis)    {        double B = col.Width, D = col.Depth;        double fck = col.Fck, fy = col.Fy;        double pt  = col.PtProvided / 100.0;        // Effective cover = clear cover + shear dia + main bar dia / 2        double dPrime = col.ClearCover + col.ShearBarDia + col.MainBarDia / 2.0;        var barLayers = BuildBarLayers(col, axis, dPrime);  // returns (y, area) pairs        var stressTable = IS456StressTable(fy);             // lookup table        var points = new List<CurvePoint>();        // Sweep neutral axis from 0.01*D to 10*D  (199 steps in original VBA)        for (int i = 0; i <= 60; i++)        {            double xu = (0.01 + i * (10.0 - 0.01) / 60.0) * D;            // IS 456 stress block: a = 0.362*xu, xbar = 0.416*xu            double cc = 0.362 * fck * B * xu;            double mc = cc * (D / 2.0 - 0.416 * xu);            double cs = 0, ms = 0;            foreach (var (y, area) in barLayers)            {                double strain = 0.0035 * (xu - y) / xu;                double fsi    = Interpolate(stressTable, strain);                cs += area * (fsi - 0.447 * fck);                ms += area * (fsi - 0.447 * fck) * (y - D / 2.0);            }            double pu = cc + cs;            double mu = Math.Abs(mc + ms);            points.Add(new CurvePoint(                PNorm: pu / (fck * B * D),                MNorm: mu / (fck * B * D * D)            ));        }        return new InteractionCurveResult        {            Points     = points,            Axis       = axis,            PtUsed     = col.PtProvided,            GeneratedAt = DateTime.UtcNow        };    }    // Find minimum Pt such that ALL combinations are SAFE    public double FindMinSafePt(WallMember col, Axis axis)    {        for (double pt = 0.10; pt <= 3.00; pt += 0.05)        {            col.PtProvided = pt;            var result = Generate(col, axis);            if (AllCombinationsSafe(col, result)) return pt;        }        return 3.00; // return max if no safe Pt found    }}WPF note:  Register InteractionCurveService as a singleton in your DI container (App.xaml.cs using Microsoft.Extensions.DependencyInjection). Persistence — Saving Data Between SessionsThe web prototype has no persistence — all data is lost on page refresh. A WPF application needs a real storage layer. Here are three options in order of complexity:8.1 Option A — JSON Files (Recommended to Start)Serialize each Project object to a JSON file in the user's AppData folder using System.Text.Json. One file per project. Simple, no database required, human-readable for debugging.// C# — Repositories/JsonProjectRepository.cspublic class JsonProjectRepository{    private readonly string _folder =        Path.Combine(Environment.GetFolderPath(            Environment.SpecialFolder.ApplicationData),            "WallDesigner");    public async Task SaveAsync(Project project)    {        var path = Path.Combine(_folder, project.Id + ".json");        var json = JsonSerializer.Serialize(project,            new JsonSerializerOptions { WriteIndented = true });        await File.WriteAllTextAsync(path, json);    }    public async Task<Project?> LoadAsync(string id)    {        var path = Path.Combine(_folder, id + ".json");        if (!File.Exists(path)) return null;        var json = await File.ReadAllTextAsync(path);        return JsonSerializer.Deserialize<Project>(json);    }    public IEnumerable<string> ListProjectIds()        => Directory.GetFiles(_folder, "*.json")                    .Select(Path.GetFileNameWithoutExtension)!;}8.2 Option B — SQLiteUse Microsoft.Data.Sqlite or Entity Framework Core + SQLite provider for a proper relational database in a single file. Better for querying across projects (e.g. "show all unsafe Walls across all projects"). More setup but scales to larger firms.// EF Core DbContext for the apppublic class DesignerDbContext : DbContext{    public DbSet<Project>       Projects       { get; set; }    public DbSet<WallMember>  WallMembers  { get; set; }    public DbSet<LoadCombination> LoadCombinations { get; set; }    protected override void OnConfiguring(DbContextOptionsBuilder opts)        => opts.UseSqlite(            "Data Source=WallDesigner.db");}8.3 Option C — Cloud / API BackendThis is the Phase 3 roadmap from the earlier conversation — move persistence to an ASP.NET Core API with a SQL Server or PostgreSQL database. Multiple engineers share projects in real time. The WPF app calls the REST API; the web app also calls the same API. This is the largest investment but enables team collaboration.OptionSetup effortMulti-userBest forA — JSON files1–2 hoursNo (one engineer per machine)Prototype, small practice, single engineerB — SQLiteHalf dayNo (one machine at a time)Growing practice, single officeC — REST API + cloud DBSeveral daysYes (web + desktop share data)Multi-engineer team, branch offices9. What Changes vs. the Web Prototype11.1 Things That Are the SameAll data models — Project, WallMember, LoadCombination, CurvePoint are identical in structure.The seven-step pipeline — the logical flow from ETABS import to Add to Summary is unchanged.The interaction curve algorithm — same IS 456 strain-compatibility sweep, just in C# instead of JavaScript.The Pt-slider interaction pattern — adjust Pt → curve recomputes → SAFE/UNSAFE updates instantly.The override-with-note pattern — engineer can record a justification before finalising an unsafe wall.11.2 Things That Are Different (Better in WPF)Persistence — WPF saves to disk; web prototype loses everything on refresh.Real file import — WPF can open the ETABS .xlsx directly with ClosedXML; web app requires a manual paste step.PDF export — WPF generates a proper formatted calculation sheet PDF; web prototype relies on browser print.Chart quality — OxyPlot/LiveCharts provide interactive zoom, tooltips, and annotation on the P-M diagram; the web SVG is hand-drawn and static.Performance on large projects — 500+ walls, heavy curve generation runs natively in C# with no JavaScript VM overhead.11.3 Things That Are Better in the Web VersionZero install — any engineer with a browser URL can use it immediately.Sharing & review — send a link, anyone sees the live interface.Cross-platform — works on Windows, Mac, Linux, iPad.Easier updates — push to GitHub Pages; all users get the new version on next refresh.12. SummaryThe web prototype and a WPF application are architecturally closer than they might appear. The fundamental difference is not how data is structured or how the engineer interacts with the tool — it is how state changes propagate to the UI and how data is saved.ConcernWeb (React) AnswerWPF (MVVM) AnswerHow UI reacts to data changesuseState setter → React re-renders the component treeINotifyPropertyChanged → XAML binding engine updates controlsHow data is passed between screensProps down / callbacks up; or top-level useStateNavigationService / Messenger + ViewModel injectionHow calculations stay out of the UIPure JavaScript functions called inside useMemo()Service classes injected into ViewModelsHow data survives the sessionIt does not (prototype only)Repository pattern — JSON files or SQLiteHow buttons trigger actionsonClick={handler} in JSXCommand={Binding SomeCommand} in XAMLThe practical next step is to start with Sprint 1 from the roadmap above — get the shell running with three navigable pages and the same mock data as the web prototype. LoadCombinations.First();    }    // ── Passthrough display properties ───────────────────────────    public string wallId    => _wall.Id;    public string FloorLevel  => _wall.FloorLevel;    public double Width       => _wall.Width;    public double Depth       => _wall.Depth;    public int    Fck         => _wall.Fck;    public int    Fy          => _wall.Fy;    // ── Adjustable: Pt ───────────────────────────────────────────    [ObservableProperty]    [NotifyPropertyChangedFor(nameof(CurvePoints))]    [NotifyPropertyChangedFor(nameof(IsSafe))]    [NotifyPropertyChangedFor(nameof(InteractionRatio))]    [NotifyPropertyChangedFor(nameof(RecommendedPt))]    private double _pt;    // ── Axis toggle ──────────────────────────────────────────────    [ObservableProperty]    [NotifyPropertyChangedFor(nameof(CurvePoints))]    [NotifyPropertyChangedFor(nameof(IsSafe))]    [NotifyPropertyChangedFor(nameof(InteractionRatio))]    private Axis _selectedAxis;    // ── Load combinations ────────────────────────────────────────    public ObservableCollection<LoadCombination> LoadCombinations { get; }    [ObservableProperty]    [NotifyPropertyChangedFor(nameof(DemandP))]    [NotifyPropertyChangedFor(nameof(DemandM))]    [NotifyPropertyChangedFor(nameof(IsSafe))]    [NotifyPropertyChangedFor(nameof(InteractionRatio))]    private LoadCombination _selectedCombo;    // ── Computed: curve & demand ─────────────────────────────────    public IEnumerable<CurvePoint> CurvePoints =>        _curveService.Generate(_wall with { PtProvided = Pt }, SelectedAxis).Points;    public double DemandP =>        _curveService.NormaliseAxial(SelectedCombo.Pu, _wall);    public double DemandM =>        _curveService.NormaliseMoment(            SelectedAxis == Axis.Major ? SelectedCombo.Mu3Final : SelectedCombo.Mu2Final,            _wall, SelectedAxis);    public double InteractionRatio =>        _curveService.InteractionRatio(CurvePoints, DemandP, DemandM);    public bool   IsSafe         => InteractionRatio <= 1.0;    public double RecommendedPt  =>        _curveService.FindMinSafePt(_wall, SelectedCombo, SelectedAxis);    // ── Override note ────────────────────────────────────────────    [ObservableProperty] private bool   _overrideDialogOpen;    [ObservableProperty] private string _overrideNote = string.Empty;    // ── Commands ─────────────────────────────────────────────────    [RelayCommand]    private void ApplyRecommendedPt() => Pt = RecommendedPt;    [RelayCommand]    private void OpenOverrideDialog()  => OverrideDialogOpen = true;    [RelayCommand]    private void SaveOverride()        => OverrideDialogOpen = false;    [RelayCommand]    private async Task AddToSummaryAsync()    {        _wall.PtProvided   = Pt;        _wall.Status       = IsSafe ? DesignStatus.Safe : DesignStatus.Unsafe;        _wall.OverrideNote = OverrideNote;        await _repo.SaveWALLAsync(_wall);        // Navigate back to ProjectView via Messenger or NavigationService    }}
